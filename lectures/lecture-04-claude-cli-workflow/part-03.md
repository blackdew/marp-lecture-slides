# 3강: 실전 워크플로우 - Claude와 함께 일하기

## 3.1 완벽한 워크플로우: 6단계

### 3.1.1 핵심 메시지

Claude CLI로 프로젝트를 진행할 때는 항상 같은 패턴을 따릅니다. 이 6단계만 기억하면 어떤 프로젝트든 성공할 수 있습니다.

### 3.1.2 6단계 워크플로우

```
1. 문서 준비 → 2. Plan 모드 대화 → 3. 계획 확정
     ↓
6. 문서 업데이트 ← 5. 작업 완료 확인 ← 4. 실행 모드 작업
```

**1단계: 문서 준비**
- docs/ 폴더에 project-plan.md, todo.md 작성
- 무엇을 만들지 대략적으로 정리

**2단계: Plan 모드로 대화**
- `claude --plan`으로 시작
- Claude와 충분히 대화하며 세부사항 확정
- 질문에 답하고, 불명확한 부분 명확히

**3단계: 계획 확정**
- Claude가 제시한 계획 검토
- "이대로 진행하자" 확인
- `accept edits on`으로 실행 모드 전환

**4단계: 실행 모드 작업**
- Claude가 파일 생성 및 코드 작성
- Cursor IDE로 진행 상황 확인
- 필요하면 추가 지시

**5단계: 작업 완료 확인**
- 만들어진 결과물 확인
- 테스트 및 검증
- 문제 있으면 즉시 수정 요청

**6단계: 문서 업데이트**
- todo.md에 완료 체크
- progress.md에 작업 내역 기록
- 다음 작업 계획

## 3.2 Plan 모드: 충분한 대화의 힘

### 3.2.1 핵심 메시지

Plan 모드에서의 충분한 대화가 프로젝트 성공의 80%를 결정합니다. 서두르지 말고, 명확해질 때까지 대화하세요.

### 3.2.2 좋은 대화 vs 나쁜 대화

**나쁜 예: 서두르는 대화**

```
You: 블로그 만들어줘.

Claude: 어떤 기능이 필요한가요?

You: 그냥 일반적인 블로그. 빨리 만들어.

Claude: (혼란스러워하며 일반적인 것 제작... 나중에 많은 수정 필요)
```

**좋은 예: 충분한 대화**

```
You: 개인 블로그를 만들고 싶어. 계획을 같이 세우자.

Claude: 좋습니다. 몇 가지 질문드릴게요:
        - 글 작성 방식은? (마크다운, 에디터)
        - 댓글 기능이 필요한가요?
        - 카테고리 분류는 어떻게 할까요?
        - 어느 플랫폼에 배포하시나요?

You: 마크다운으로 글을 작성하고, 댓글은 일단 필요 없어.
     카테고리는 태그 방식으로 하고, Vercel에 배포할 거야.
     그리고 다크모드도 있었으면 좋겠어.

Claude: 완벽합니다. 추가로:
        - RSS 피드가 필요하신가요?
        - 검색 기능은 어떠신가요?
        - 이미지 최적화는 자동으로 할까요?

You: RSS는 필요하고, 검색은 나중에 추가할게.
     이미지 최적화는 좋겠다.

Claude: 네, 그럼 다음과 같이 계획을 세웠습니다:
        [상세한 계획 제시...]
        이대로 진행할까요?

You: 완벽해! 시작하자.
```

### 3.2.3 좋은 대화의 원칙

1. **서두르지 않기:** 계획 단계에서는 시간을 충분히 투자
2. **구체적으로 답하기:** "그냥", "적당히" 같은 애매한 표현 피하기
3. **추가 아이디어 환영:** 대화 중 떠오르는 아이디어 즉시 공유
4. **이해 확인:** Claude의 제안을 제대로 이해했는지 확인

- `img_p3_4_planning_time.svg`: "계획 30분 → 실행 10분" vs "계획 5분 → 실행 1시간 + 수정 2시간"

## 3.3 세션 분리 전략: 작업을 나누어 진행하기

### 3.3.1 핵심 메시지

큰 프로젝트는 한 번에 하지 말고, 작은 작업 단위로 나누어 세션별로 진행합니다. 각 세션마다 문서를 업데이트하면 프로젝트가 명확하게 관리됩니다.

### 3.3.2 세션 분리의 장점

**장점 1: 집중력 유지**
- 한 세션에 한 가지 작업만 집중
- 지치지 않고 효율적으로 작업

**장점 2: 명확한 진행 상황**
- 각 세션 후 문서 업데이트로 진행 상황 명확
- "어디까지 했더라?" 고민 불필요

**장점 3: 오류 최소화**
- 작은 단위로 테스트하고 확인
- 문제 발생 시 범위가 작아 해결 쉬움

**장점 4: 유연한 일정**
- 시간 날 때마다 한 세션씩 진행 가능
- 중간에 멈춰도 문서로 이어서 시작

### 3.3.3 세션 분리 예시

**프로젝트: 전자상거래 웹사이트**

```markdown
## 세션 1 (2025-10-17)
**목표:** 프로젝트 초기 설정 및 메인 페이지
**작업:**
- [x] Next.js 프로젝트 생성
- [x] 폴더 구조 설정
- [x] 메인 페이지 레이아웃
**결과:** 기본 구조 완성, localhost에서 확인 완료

## 세션 2 (2025-10-18)
**목표:** 상품 목록 페이지
**작업:**
- [x] 상품 데이터 모델 설계
- [x] 상품 목록 컴포넌트
- [x] 상품 카드 UI
**결과:** 상품 목록 표시 기능 완성

## 세션 3 (2025-10-19)
**목표:** 상품 상세 페이지
**작업:**
- [ ] 상품 상세 정보 표시
- [ ] 이미지 갤러리
- [ ] 장바구니 담기 버튼
**상태:** 진행 예정
```

### 3.3.4 세션 시작 전 체크리스트

- [ ] docs/todo.md에서 이번 세션의 작업 확인
- [ ] Claude CLI 시작: `claude --plan`
- [ ] 이전 세션 작업 내용 Claude에게 공유
- [ ] 이번 세션 목표 명확히 전달

### 3.3.5 세션 종료 전 체크리스트

- [ ] 작업 결과 테스트 및 확인
- [ ] todo.md 업데이트 (완료 체크)
- [ ] progress.md에 작업 내역 기록
- [ ] 다음 세션 작업 todo.md에 추가
- [ ] git commit으로 작업 저장

## 3.4 문서 업데이트: 작업의 마무리

### 3.4.1 핵심 메시지

작업이 끝나면 반드시 문서를 업데이트합니다. 문서 업데이트를 잊으면 나중에 혼란이 생깁니다.

### 3.4.2 업데이트해야 할 3가지 문서

**1. todo.md 업데이트**

```markdown
## 완료
- [x] 로그인 페이지 UI 구현 (2025-10-17 완료)
  - 이메일/비밀번호 입력 폼
  - 로그인 버튼
  - 회원가입 링크

## 다음 작업
- [ ] 로그인 API 연동
- [ ] 세션 관리 구현
```

**2. progress.md 업데이트**

```markdown
## 2025-10-17 세션 3

### 완료한 작업
- 로그인 페이지 UI 완성
- React Hook Form으로 폼 관리
- 기본 유효성 검사 추가

### 발견한 문제
- 비밀번호 필드에 보기/숨기기 토글 필요
- 소셜 로그인 버튼 위치 조정 필요

### 다음 세션 계획
- 로그인 API 연동
- 토큰 저장 및 세션 관리
```

**3. milestones.md 업데이트 (필요 시)**

```markdown
## 🎯 마일스톤 1: MVP
**상태:** 진행 중 (70%)

**완료:**
- [x] 프로젝트 설정
- [x] 메인 페이지
- [x] 로그인 UI

**남은 작업:**
- [ ] 로그인 기능
- [ ] 회원가입 기능
```

### 3.4.3 Claude에게 문서 업데이트 요청하기

```
You: 작업이 완료됐어. todo.md와 progress.md를 업데이트해줘.
     오늘 완료한 작업은 "로그인 페이지 UI 구현"이고,
     다음에 할 일은 "로그인 API 연동"이야.

Claude: 네, 문서를 업데이트하겠습니다.
        [todo.md와 progress.md 자동 업데이트]
        업데이트 완료했습니다. 확인해주세요.
```

## 3.8 실전 사례 4: 범용 가이드 - 모든 프로젝트에 적용

### 3.8.1 보편적인 원칙

어떤 프로젝트든 다음 원칙을 따르면 성공할 수 있습니다:

**원칙 1: 문서 먼저**
- 코드보다 문서를 먼저 작성
- 무엇을 만들지 명확히 정의

**원칙 2: 작게 나누기**
- 큰 프로젝트를 작은 작업으로 분할
- 한 번에 한 가지만 집중

**원칙 3: 자주 확인**
- 각 작업 후 테스트
- 문제 조기 발견 및 수정

**원칙 4: 기록 남기기**

## 3.9 문제 해결: 자주 묻는 질문

### 3.9.1 Q1: Claude가 내 의도를 잘못 이해했어요

**해결책:**

```
You: 잠깐, 내가 원한 건 그게 아니야.
     다시 설명할게. 내가 원하는 건...

Claude: 아, 이해했습니다. 다시 진행하겠습니다.
```

- 언제든 멈추고 수정 가능
- "plan 모드로 돌아가자"고 요청

### 3.9.2 Q2: 세션을 나누면 Claude가 이전 내용을 기억할까요?

**해결책:**
- 문서가 기억 역할
- 새 세션 시작 시:

```
You: docs/project-plan.md와 docs/progress.md를 읽어봐.
     지난번에 우리가 로그인 페이지를 완성했어.
     이번에는 회원가입 기능을 만들자.

Claude: 문서를 확인했습니다. 이어서 진행하겠습니다.
```

### 3.9.3 Q3: 문서를 매번 업데이트하기 귀찮아요

**해결책:**
- Claude에게 자동 업데이트 요청
- 습관이 되면 5분도 안 걸림
- 나중에 혼란 겪는 것보다 훨씬 이득

### 3.9.4 Q4: 프로젝트가 너무 커서 어디서부터 시작할지 모르겠어요

**해결책:**

```
You: 이 프로젝트가 너무 커서 압도되는데,
     어떻게 시작하면 좋을까?

Claude: MVP(최소 기능 제품)부터 시작하는 게 좋겠습니다.
        가장 핵심적인 기능 3가지만 먼저 만들고,
        그 다음에 추가 기능을 더하는 방식으로 진행할까요?

You: 좋아, 핵심 기능 3가지를 정해보자.
```

## 3.10 정리: 3강 핵심 요약

### 핵심 메시지 5가지

1. **6단계 워크플로우를 따르세요:** 문서 준비 → Plan 대화 → 확정 → 실행 → 확인 → 업데이트
2. **Plan 모드에서 충분히 대화하세요:** 서두르지 말고 명확해질 때까지
3. **세션을 분리하세요:** 큰 프로젝트는 작은 작업으로 나누어 진행
4. **항상 문서를 업데이트하세요:** 작업 후 todo, progress 문서 업데이트
5. **어떤 프로젝트든 같은 원칙:** 문서 먼저, 작게 나누기, 자주 확인, 기록 남기기

**시작 전:**
- [ ] 문서 준비
- [ ] Plan 모드 시작
- [ ] 충분한 대화

**진행 중:**
- [ ] 한 번에 한 가지만
- [ ] 자주 확인
- [ ] Cursor로 결과 확인

**완료 후:**
- [ ] 문서 업데이트
- [ ] git commit
- [ ] 다음 계획
