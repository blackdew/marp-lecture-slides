## 실습 준비: CLI 환경 이해하기

### CLI(Command Line Interface)란?

**CLI**: 텍스트 명령어로 컴퓨터와 대화하는 방식
- GUI (그래픽): 마우스 클릭, 아이콘
- CLI (명령어): 키보드로 텍스트 입력

**왜 CLI를 사용할까요?**
- 빠르고 정확한 작업
- 반복 작업 자동화
- 개발 도구 대부분이 CLI 기반

### 터미널 여는 법

**Mac**: Spotlight 검색 (⌘+Space) → "터미널" 입력
**Windows**: 시작 메뉴 → "cmd" 또는 "PowerShell" 검색

## AI Agent란?

<div style="display: grid; grid-template-columns: 1fr 1fr; gap: 40px;">

<div>

### 챗봇 vs Agent - 무엇이 다를까요?

| 구분 | 챗봇 | Agent |
|------|------|-------|
| **할 수 있는 것** | 대화, 조언, 설명 | 대화 + **파일 조작** |
| **도구 사용** | 없음 | 파일 읽기/쓰기, 코드 실행 |
| **비유** | 똑똑한 친구 | **도구를 쓰는 비서** |
| **예시** | "Python 코드 알려줘" | "Python 파일 만들어줘" |

</div>

<div>

### CLI를 사용하는 이유

**CLI를 통해 Agent가 컴퓨터 기능에 직접 접근할 수 있습니다**

- **파일 시스템**: 내 컴퓨터의 파일을 읽고, 만들고, 수정
- **명령 실행**: 프로그램 실행, 데이터 처리, 작업 자동화
- **도구 연결**: Git, Python, 데이터베이스 등 개발 도구 활용

**웹 챗봇과의 차이**:
- 웹 챗봇: 브라우저 안에서만 작동 (복사/붙여넣기 필요)
- CLI Agent: 내 컴퓨터에서 직접 작업 (자동화 가능)

</div>

</div>

<div class="highlight">
핵심: CLI를 사용하면 Agent가 <strong>내 컴퓨터의 도구(Tools)를 직접 사용</strong>할 수 있습니다!
</div>

---

## Agent의 구조: Tools의 힘

### Agent = LLM + Memory + Planning + **Tools**

**AI Agent를 구성하는 4가지 요소:**

**1. LLM (뇌)**
- 언어 이해 및 생성

**2. Memory (기억)**
- 대화 맥락 유지

**3. Planning (계획)**
- 단계별 작업 계획

**4. Tools (도구) ← 핵심!**
- Read, Write, Edit, Bash...
- 실제 작업을 수행하는 능력

**비유**: 요리사에게 레시피만 vs **재료와 도구까지 준 것**

## Plan 모드란?

### 안전장치: 실행 전에 계획 검토하기

**Plan 모드**: Agent가 실행하기 **전에** 계획을 보여줌

| 일반 모드 | Plan 모드 |
|----------|-----------|
| 프롬프트 → 바로 실행 | 프롬프트 → **계획 확인** → 승인 → 실행 |
| 빠르지만 위험 | 느리지만 안전 |

**비유**: "일 시키기 전에 보고받기"

### Plan 모드 진입 방법

**Claude Code**: 키보드에서 `Shift + Tab` 두 번 누르기
**Gemini, Codex**: Plan 모드 미지원 (일반 모드에서 신중하게 프롬프트 작성)

**표시**: `[Plan Mode]` 라는 표시가 나타남 (Claude만)

# 1. 선택 (Choice)

왜 AI 시대에 선택이 더 어려워졌는가?

---

<div class="question">
AI 시대의 역설: 더 많은 것을 할 수 있는데, 왜 결정은 더 어려워졌을까요?
</div>

### 과거 vs 현재

| 과거 | AI 시대 |
|------|---------|
| 정보 부족 | **정보 과잉** |
| 선택지 제한됨 | **무한한 선택지** |
| 실행이 어려움 | **실행은 쉬움** |
| 결정이 상대적으로 쉬움 | **결정이 더 어려움** |

**역설: 더 많은 선택지 = 더 어려운 선택**

---

## 1-1. 구체적 경험: 같은 질문, 다른 답변

### 여러분의 경험

Part 2에서 이런 경험 하셨나요?

- "예쁘게 만들어줘" → 내가 원한 스타일이 아님
- "정리해줘" → 정리 기준이 내 생각과 다름
- "개선해줘" → 뭘 개선했는지 모르겠음

### 같은 지시를 다시 하면?

**"예쁘게 만들어줘"** → 또 다른 스타일...

**왜 이런 일이 생길까요?**

---

## 1-2. LLM의 작동 원리: 확률 기반 추론

### LLM은 어떻게 작동하는가?

**"오늘 날씨가"** 다음에 올 단어는?

- "좋아요" (40%)
- "맑아요" (25%)
- "흐려요" (20%)
- "나빠요" (15%)

**핵심:** LLM은 **확률로 다음 단어를 예측**합니다

---

## 1-3. 확률 기반 추론이 의미하는 것

<div class="highlight">
정답이 없고, 여러 선택지가 있을 때 Agent는 확률로 선택합니다
</div>

### 예시: "보고서 써줘"

Agent가 직면하는 무수한 선택들:
- 누구를 위한 보고서? → 10가지 가능성
- 어떤 형식? → 20가지 가능성
- 어떤 톤? → 15가지 가능성
- 어떤 길이? → 무한한 가능성

**→ 10 × 20 × 15 × ∞ = 무한한 조합**

### 결과

매번 다른 보고서가 나올 수 있고, 당신이 원하는 것과 다를 확률이 높음

---

## 1-4. 그래서 선택은 당신의 몫

| Agent의 역할 | 당신의 역할 |
|-------------|-----------|
| **정보 수집** - 여러 옵션 조사 | **가치 판단** - "우리 팀엔 이게 중요해" |
| **비교 분석** - 장단점 표로 정리 | **최종 결정** - "A로 하자!" |
| **추천** - "이런 경우엔 이게 좋아요" | **책임** - "내가 선택한 거니까" |

<div class="highlight">
Agent는 조언자, 당신은 결정자
</div>

---

## 1-5. 실습: Agent에게 선택지 요청하기

### 시나리오

주간 보고서를 어떤 형식으로 쓸까?

### 프롬프트 예시

```
주간 업무 보고서를 작성하려고 합니다.
다음 형식을 비교해주세요:

1. 불릿 포인트 형식 (간결)
2. 스토리텔링 형식 (상세)
3. 표 형식 (수치 중심)

각 형식의 장단점을 표로 정리하고,
경영진에게 보고하는 상황에 어떤 게 적합한지 추천해주세요.
```

---

## 1-6. Agent의 답변 예시

| 형식 | 장점 | 단점 | 적합한 상황 |
|------|------|------|-----------|
| 불릿 포인트 | 빠른 파악, 명확 | 맥락 부족 | 정기 보고 |
| 스토리텔링 | 이해 쉬움, 설득력 | 시간 많이 걸림 | 프로젝트 완료 보고 |
| 표 형식 | 수치 비교 쉬움 | 지루함 | 성과 분석 |

**추천:** 경영진 보고는 불릿 포인트 + 핵심 수치 조합

**→ 최종 결정은 여러분이 하세요!**

---

## 1-7. 좋은 질문 패턴

### 나쁜 질문 vs 좋은 질문

| 나쁜 질문 | 문제점 | 좋은 질문 |
|----------|--------|----------|
| "뭐가 좋아?" | 기준 없음 | "A와 B의 장단점을 표로 비교해줘" |
| "추천해줘" | 상황 불명확 | "내 상황은 X야. A와 B 중 뭐가 적합할까?" |
| "정답 알려줘" | 정답 없음 | "각 선택지의 사용 사례를 알려줘" |

**핵심:** 선택지를 요청하되, 결정은 내가 한다

---

## 1-8. 선택 키워드 정리

<div class="highlight">
AI 시대의 역설: 더 많은 선택지 → 더 어려운 선택 → 더 큰 책임
</div>

### AI가 바꾼 것

- ✅ 실행 장벽 낮아짐 (누구나 할 수 있음)
- ✅ 선택지 무한대로 증가
- ❌ 선택의 어려움은 그대로 (오히려 증가)

### 우리의 역할

1. **선택지 요청**: "A와 B를 비교해줘"
2. **장단점 분석**: 표로 정리 요청
3. **최종 선택과 책임**: 내가 선택하고 책임진다

**다음:** 계획 (왜 Agent는 엉뚱한 결과를 낼까?)

# 2. 계획 (Planning)

왜 Agent는 때때로 엉뚱한 결과를 낼까요?

---

<div class="question">
상황: "파일 정리해줘"라고 했더니, Agent가 중요한 파일을 삭제했습니다. 왜 그럴까요?
</div>

### 여러분의 경험

- "메일 써줘" → 완전히 다른 톤으로 작성됨
- "데이터 정리해줘" → 필요한 열을 삭제함
- "요약해줘" → 핵심을 놓침

**공통점: 지시가 모호했다**

---

## 2-1. 컨텍스트의 중요성

<div class="highlight">
컨텍스트 = Agent의 '단기 기억' (대화의 맥락)
</div>

### 비어있는 컨텍스트의 문제

**"파일 정리해줘"**

Agent가 추론해야 하는 것들:
- 어떤 파일? → 추론으로 채움
- 어떤 기준? → 추론으로 채움
- 어디에? → 추론으로 채움
- 어떻게? → 추론으로 채움

**→ 예측 불가능한 결과**

---

## 2-2. 좋은 계획의 3원칙

| 원칙 | 나쁜 예 | 좋은 예 |
|------|--------|--------|
| **1. 구체성** | "파일 정리해줘" | "다운로드 폴더의 PDF를 년도별로 정리" |
| **2. 단계성** | "데이터 분석해줘" | "1단계: CSV 로드<br>2단계: 중복 제거<br>3단계: 집계" |
| **3. 결과물 명시** | "요약해줘" | "3-5문장으로 요약해서 summary.md로 저장" |

<div class="highlight">
구체적으로 말할수록 원하는 결과가 나옵니다
</div>

---

## 2-3. 나쁜 프롬프트 vs 좋은 프롬프트

### 나쁜 프롬프트

```
보고서 써줘
```

**문제:** 모든 것을 Agent가 추론해야 함

### 좋은 프롬프트

```
주간 판매 데이터를 기반으로 경영진용 요약 보고서 작성.

1단계: sales_jan.csv, sales_feb.csv, sales_mar.csv 로드
2단계: 월별 총 매출, 담당자별 매출, 증감률 추출
3단계: 간결하고 수치 중심으로 summary.md 파일로 저장

형식: 마크다운, 1페이지 이내, 불릿 포인트
```

---

## 2-4. 실습: 프롬프트 개선하기 (1분 생각)

### 나쁜 프롬프트

```
메일 써줘
```

**여러분이 개선해보세요:**

- 누구에게?
- 무엇에 대해?
- 어떤 톤으로?
- 어떤 형식으로?
- 결과물은?

---

## 2-5. 답안 예시

### 개선된 프롬프트

```
회의록(meeting_notes.md)을 기반으로 팀원들에게 보낼 메일 작성.

1단계: meeting_notes.md 파일 읽기
2단계: 주요 안건, 결정 사항, 액션 아이템 추출
3단계: 친근하고 간결한 톤으로 메일 초안 작성
4단계: email_to_team.md 파일로 저장

형식: 인사 → 회의 요약 → 액션 아이템 → 마무리
길이: 10줄 이내
```

**적용된 3원칙:** ✓ 구체성  ✓ 단계성  ✓ 결과물 명시

---

## 2-6. Part 2 실습 다시 보기

### Part 2에서 했던 프롬프트

```
간단한 카운터 앱을 만들어줘.
버튼을 누르면 숫자가 올라가고, 리셋 버튼도 추가해줘.
```

### 3원칙 적용 여부

- **구체성** ✓ 카운터, 증가 버튼, 리셋 버튼
- **단계성** ✗ (간단해서 생략 가능)
- **결과물** ✓ HTML 파일

**→ 간단한 작업은 구체성만 있어도 충분!**

## 2-7. 언제 상세하게, 언제 간단하게?

# 3. 문서화 (Documentation)

AI 시대에도 문서화가 필요할까요?

---

<div class="question">
질문: AI가 다 기억하는데, 왜 문서를 만들어야 하나요?
</div>

### 여러분의 생각

- "Agent에게 물어보면 되잖아요?"
- "문서 쓸 시간에 다른 일 하는 게 낫지 않나요?"
- "귀찮은데 꼭 해야 하나요?"

**정말 필요 없을까요?**

---

## 3-1. 전통적인 문서화의 문제

### 왜 문서화는 실패하는가?

| 문제 | 설명 |
|------|------|
| ⏰ **시간 부족** | 업무도 바쁜데 문서까지? |
| 💰 **비용 과다** | 문서 작성 = 작업 시간의 30-50% |
| 📉 **우선순위 밀림** | 결과물이 먼저, 문서는 나중에 |
| 😫 **작성 귀찮음** | 반복적이고 지루한 작업 |

### 결과

→ 인수인계 실패, 맥락 손실, 중복 작업, 품질 저하

---

## 3-2. Agent가 바꾸는 문서화의 경제학

<div class="highlight">
Agent는 문서화의 경제학을 완전히 바꿨습니다
</div>

| Before AI | After AI |
|-----------|----------|
| 문서는 나중에 | **문서가 먼저** |
| 시간 없음 | Agent가 자동 생성 |
| 사람이 작성 | **Agent 작성 → 사람 검토** |
| 문서화 = 비용 | **문서화 = 거의 무료** |

**핵심 변화:** 이제는 비용 때문이 아니라 **비전 때문에** 문서를 만듭니다

---

## 3-3. 문서화의 새로운 패러다임

### 전통적인 업무 흐름

1. 작업 수행
2. 결과물 생성
3. (시간 있으면) 문서 작성 ← **보통 생략됨**

### AI 시대 업무 흐름

1. **문서 먼저 작성** (Agent에게 요청)
2. 작업 수행 (Agent 실행)
3. **문서 자동 업데이트** (Agent에게 요청)

**결과:** 문서가 항상 최신 상태로 유지됨

---

## 3-4. 실습: README 만들기

### 시나리오

Part 2에서 만든 파일들을 정리하는 README 작성

### 프롬프트

```
지금까지 만든 파일들(welcome.html, counter.html 등)을
설명하는 README.md를 만들어줘.

포함 내용:
- 프로젝트 개요
- 파일 목록 및 설명
- 각 파일 사용 방법
- 배운 내용 요약

형식: 마크다운, 읽기 쉽게
```

---

## 3-5. Agent가 할 일 vs 내가 할 일

### Agent의 역할

1. 현재 디렉토리 파일 확인
2. 각 파일 내용 분석
3. README.md 자동 생성

### 내 역할

1. 생성된 README.md 검토
2. 빠진 내용 추가 요청
3. 톤 조정 요청
4. 최종 승인

<div class="highlight">
Agent는 초안, 나는 검토 및 개선
</div>

---

## 3-6. 문서화 실무 활용 시나리오

| 상황 | 프롬프트 | 결과 |
|------|---------|------|
| **프로젝트 인수인계** | "모든 파일 분석해서 인수인계 문서 만들어줘" | 30분 만에 완성 |
| **회의록 정리** | "오늘 회의 내용 요약해서 공유 문서로" | 회의 직후 즉시 공유 |
| **작업 이력 관리** | "이번 달 작업 내용을 월간 리포트로" | 보고서 자동 생성 |

### 즉석 도구 만들기의 원칙

**1. 완벽함보다 작동하는 것**
- 첫 버전은 10분 안에
- 작동하면 성공

**2. 디자인보다 기능 우선**
- 예쁜 것은 나중에
- 기능이 먼저

- 기본 → 테스트 → 개선
- 한 번에 완벽하게 만들려 하지 말기

**4. 구체적인 예시 제공**
- "버튼" → "큰 파란색 버튼"
