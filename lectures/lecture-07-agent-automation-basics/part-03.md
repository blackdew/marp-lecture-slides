```markdown
# 비어있는 컨텍스트의 문제

## 예시 1: 모호한 지시

"웹사이트 만들어줘"

**비어있는 컨텍스트:**
- 어떤 기술 스택?
- 어떤 기능?
- 어떤 디자인?
- 누구를 위한?

**결과:** 매번 다른 웹사이트

## 예시 2: 불완전한 정보

"함수 리팩토링해줘"

**비어있는 컨텍스트:**
- 어떤 함수?
- 왜 리팩토링?
- 어떤 부분을?

**결과:** 의도와 다른 수정

---
```

## 슬라이드 4: 나쁜 계획 vs 좋은 계획

```markdown
# 나쁜 계획 vs 좋은 계획

| 나쁜 계획 | 좋은 계획 |
|----------|----------|
| "웹사이트 만들어줘" | "React로 할일 관리 웹앱 만들어줘. 기능: 추가/삭제/완료. 스타일: 미니멀. 저장: localStorage" |
| "보고서 작성해줘" | "주간 보고서 작성. 포함: 1) 완료 작업 2) 다음 주 계획 3) 이슈. 형식: 마크다운, 1페이지" |
| "코드 리팩토링" | "auth.js 로그인 함수 리팩토링. 목표: 1) 에러 핸들링 2) async/await 3) 주석. API 호환성 유지" |

**차이점:** 구체성, 단계, 검증 가능성

---
```

## 슬라이드 8: 좋은 계획의 조건

```markdown
# 좋은 계획의 세 가지 조건

## 1. 구체성 (Specificity)

모호한 표현 제거, 명확한 지시

❌ "예쁘게 만들어줘"
✅ "미니멀 디자인, 흰색 배경, 파란색 버튼"

## 2. 단계성 (Step-by-step)

큰 작업을 작은 단위로 분해

❌ "프로젝트 완성해줘"
✅ "1) 파일 구조 2) 기본 기능 3) 스타일 4) 테스트"

## 3. 검증성 (Verifiable)

결과를 확인할 수 있는 기준

❌ "잘 동작하게 해줘"
✅ "버튼 클릭 시 항목 추가되고, 삭제 버튼 동작해야 함"

---
```
