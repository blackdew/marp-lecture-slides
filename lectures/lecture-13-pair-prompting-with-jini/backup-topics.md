# Backup Topics for Beat 4

청중이 강의 주제를 안 던질 때 사용할 백업. *완성된 outline*이 아니라 **최소한의 씨앗** — 라이브에서 진짜로 채울 수 있을 정도만.

## A. AI 시대의 학습법 — 메타인지를 위한 도구 사용 ⭐ 메인 백업

**한 줄 메시지**:
> 단순 정보 흡수가 아니라 *자기가 뭘 모르는지 자각하는 능력*이 진짜 학습. AI를 끄덕임 기계가 아닌 *blind spot 거울*로 쓰면 학습이 가속된다.

**예상 1시간 흐름** (라이브에서 채울 6 beat 골격):

1. 왜 학습이 어려운가 — Dunning-Kruger, sycophancy 함정
2. 메타인지란 무엇인가
3. AI를 메타인지 도구로 쓰는 법 — *"내가 뭘 모르지?"* 묻기 패턴
4. 학습 루프 닫기 — 노트, 회고, 다시 묻기
5. 실습: 한 주제로 메타인지 학습 루프 한 바퀴
6. 정리 + 일상 적용

**라이브 시드 키워드**: 메타인지, Dunning-Kruger, Feynman technique, blind spot, sycophancy, 학습 회고, 노트 시스템(Obsidian/gyeol), 자기 평가, AI 튜터링의 한계

**왜 백업 메인으로 좋은가**: 오늘 강의 메시지와 100% 정합. outline을 만드는 과정 자체가 오늘 메시지를 *재차 시연*함. 청중에게 *"여러분, 우리가 지금 만드는 이 outline이 사실 오늘 강의의 심화판이에요"* 라고 자연스럽게 연결 가능.

## B. 개인 프로젝트를 Spec-driven으로 만들기 ⭐ 보조 백업

**한 줄 메시지**:
> 코드 먼저 짜지 말고 *명세*부터. 명세를 AI가 코드로 변환하고, 결과를 보고 명세를 갱신하는 사이클이 개인 프로젝트를 *지속 가능*하게 만든다.

**예상 1시간 흐름** (라이브에서 채울 6 beat 골격):

1. 왜 개인 프로젝트는 중간에 멈추는가
2. Spec-driven이란? — 명세 → 코드 → 명세 사이클
3. 명세 작성하는 법 — README first, 사용자 시나리오, 검증 가능한 결과
4. AI에게 위임하기 — Claude Code Plan 모드, skill 활용
5. 실습: 작은 프로젝트 하나 spec으로 정의해보기
6. 정리 + 첫 spec 한 줄 쓰기

**라이브 시드 키워드**: Spec-driven, README-first, Claude Code Plan 모드, `CLAUDE.md`(project instructions), skill files, 검증 가능성, AI 위임 후 검수, contract testing, 결과 → 명세 갱신

**왜 백업 보조로 좋은가**: 슬라이드 4의 spec-driven 키워드를 *재방문 + 깊이*. 실용성 높아 컴공 학생 즉시 공감.
