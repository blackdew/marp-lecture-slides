## ⚠️ 개선 가능한 점

### 1. CLAUDE.md 예시가 너무 추상적
**문제점:**
```markdown
## CLAUDE.md 구조 예시

```markdown
# 프로젝트 컨텍스트

## 디렉토리 구조
[프로젝트 구조 설명]

## 코딩 규칙
- 네이밍 컨벤션
- 들여쓰기 규칙
- 금지 패턴
```
```
- **실제 내용**이 아니라 **템플릿**만 제공
- "200-500줄 권장"이라고 하지만 구체적 예시 없음

**개선안:**
**실제 사용 가능한 CLAUDE.md 전체를 보여주세요:**
```markdown
## 실제 예시: React 프로젝트 CLAUDE.md

[전체 내용 200줄 정도 실제로 작성]

---

## 다른 예시: Python 데이터 분석 프로젝트

[전체 내용 150줄 정도]
```

이 강의 자료 저장소의 `CLAUDE.md`를 예시로 사용하는 것도 메타적으로 좋음

### 2. 컨텍스트 관리 명령어 설명 부족
**문제점:**
```markdown
| `/context` | 토큰 확인 | 수시로 |
| `/clear` | 세션 초기화 | 150K 이상 |
| `/catchup` | 요약 복원 | `/clear` 후 |
```
- 각 명령어가 **정확히** 무엇을 하는지 불명확
- `/catchup`이 어떻게 요약을 복원하는지?
- 150K는 왜 기준인지?

**개선안:**
```markdown
## 컨텍스트 관리 명령어 상세

### `/context`
현재 대화의 토큰 사용량 확인
```
Token usage: 45231/200000
```

### `/clear`
**효과:** 대화 히스토리 전체 삭제
**주의:** 복원 불가능! 반드시 `/catchup` 고려

### `/catchup`
**효과:** 이전 대화를 자동으로 요약해서 복원
**언제:** `/clear` 직후, 컨텍스트는 유지하면서 토큰 절약

**권장 워크플로우:**
```
/context → 150K 확인 → /clear → /catchup
```
```

### 3. MCP 서버 설계 원칙이 너무 간략
**문제점:**
```markdown
## MCP 서버 설계 원칙

**최소 책임 원칙**
| MCP 역할 | 에이전트 역할 |
|----------|--------------|
| 인증 토큰 관리 | 비즈니스 로직 |
```
- "최소 책임"의 구체적 기준은?
- 어디까지가 MCP 역할이고 어디부터가 Agent 역할인지 모호

**개선안:**
**Before/After 예시:**
```markdown
## ❌ 나쁜 예: 복잡한 MCP

```python
def get_user_summary(token, user_id):
    # MCP가 너무 많은 일을 함
    data = api_call(token, user_id)
    processed = clean_data(data)
    summary = generate_summary(processed)
    recommendations = ai_recommend(summary)
    return {
        "summary": summary,
        "recommendations": recommendations
    }
```

## ✅ 좋은 예: 간결한 MCP

```python
def get_user_data(token, user_id):
    # MCP는 인증과 호출만
    headers = {"Authorization": f"Bearer {token}"}
    response = requests.get(f"{API}/users/{user_id}", headers=headers)
    return response.json()  # 원본 그대로 반환

# Agent가 데이터 처리:
# - clean_data()
# - generate_summary()
# - ai_recommend()
```
```

## 💡 우수 사례

### 1. "왜"를 설명
```markdown
## `/compact` 회피 이유
- 정보 손실 위험
- 예측 불가능한 동작
- 디버깅 어려움
```
→ "하지 마세요"만이 아니라 **이유** 설명

### 2. 대안 제시
```markdown
**전통적 접근 (✗)**
- REST API 전체 미러링
- 복잡한 비즈니스 로직 포함

**현대적 접근 (✓)**
- 간결한 게이트웨이 역할
- 인증, 보안만 처리
- 나머지는 에이전트에 위임
```
→ 나쁜 것과 좋은 것을 명확히 대비

### 3. 미래 지향적
```markdown
## 미래 전망

**발전 방향**
- 더 강력한 컨텍스트 관리
- 멀티 에이전트 협업
- 자율적 프로젝트 관리

**준비 사항**: 기본기 탄탄히, 자동화 습관화
```
→ 현재만이 아니라 앞으로의 발전 방향 제시
